Java-Maven REST API + Docker + Pipeline
Build the Spring Boot REST API, containerize it, connect to Azure SQL via Private Endpoint, and automate builds through GitHub Actions.

Phase 1
Java REST API - Spring Boot
~4 hours

1.1
Use the existing Spring Boot backend
The repo already contains the Java 21 Maven API in `backend/`. Do not recreate it with Spring Initializr; keep the existing Burger Builder controllers, tests, SQL scripts, and Maven project, then add only the deployment and observability pieces needed for Azure.

1.2
Configure the existing `application-azure.properties` for Azure SQL via JDBC
File: `backend/src/main/resources/application-azure.properties`

```properties
server.port=8080

spring.datasource.url=${DB_URL:jdbc:sqlserver://${DB_HOST:your-server.database.windows.net}:${DB_PORT:1433};database=${DB_NAME:burgerbuilder};encrypt=true;trustServerCertificate=false;hostNameInCertificate=*.database.windows.net;loginTimeout=30;}
spring.datasource.username=${DB_USERNAME:your-username}
spring.datasource.password=${DB_PASSWORD:your-password}
spring.datasource.driver-class-name=${DB_DRIVER:com.microsoft.sqlserver.jdbc.SQLServerDriver}

spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=false
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.SQLServerDialect
spring.jpa.properties.hibernate.format_sql=false

app.cors.allowed-origins=${CORS_ALLOWED_ORIGINS:http://localhost:3000,http://localhost:5173}

management.endpoints.web.exposure.include=health,info,metrics
management.endpoint.health.show-details=always
management.health.defaults.enabled=true
```

Keep the repo's `.properties` files. Do not add a parallel `application.yml`.

Also update `BurgerBuilderApplication.java` so CORS is environment-driven instead of `allowedOrigins("*")`. In production, requests should come from the App Gateway origin or the custom domain origin passed through deployment configuration.

1.3
Use the existing Burger Builder API and health route

- `backend/src/main/java/com/burgerbuilder/controller/HealthController.java` -> `GET /api/health`
- `backend/src/main/java/com/burgerbuilder/controller/IngredientController.java` -> `GET /api/ingredients`
- `backend/src/main/java/com/burgerbuilder/controller/CartController.java` -> `POST /api/cart/items`, `GET /api/cart/{sessionId}`
- `backend/src/main/java/com/burgerbuilder/controller/OrderController.java` -> `POST /api/orders`, `GET /api/orders/history`

The existing `/api/health` endpoint is what App Gateway probes. For the end-to-end database demo, use `/api/ingredients` and `/api/cart/items`.

1.4
Add Application Insights with the current Azure Java approach
Use runtime attach instead of the deprecated Spring Boot starter path.

```xml
<dependency>
  <groupId>com.microsoft.azure</groupId>
  <artifactId>applicationinsights-runtime-attach</artifactId>
  <version>3.7.8</version>
</dependency>
<dependency>
  <groupId>com.microsoft.azure</groupId>
  <artifactId>applicationinsights-core</artifactId>
  <version>3.7.8</version>
</dependency>
```

Call `ApplicationInsights.attach()` before `SpringApplication.run(...)` when `APPLICATIONINSIGHTS_CONNECTION_STRING` is present, and configure the connection string at runtime.

Phase 2
Dockerize the Backend
~1 hour

Add `backend/Dockerfile` and keep it as a multi-stage Java 21 build:

```dockerfile
FROM maven:3.9-eclipse-temurin-21-alpine AS builder
WORKDIR /app
COPY pom.xml ./
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

FROM eclipse-temurin:21-jdk-alpine
WORKDIR /app
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=builder /app/target/*.jar /app/app.jar
COPY docker-entrypoint.sh /app/docker-entrypoint.sh
RUN chmod +x /app/docker-entrypoint.sh && chown -R appuser:appgroup /app
USER appuser

EXPOSE 8080
ENTRYPOINT ["/app/docker-entrypoint.sh"]
```

Test locally:

```bash
cd backend
docker build -t backend:local .
docker run -p 8080:8080 --env-file ../environment.env backend:local
curl http://localhost:8080/api/health
```

Phase 3
Backend GitHub Actions Pipeline
~3 hours

Add `.github/workflows/backend.yml`.

Requirements:

- Run build and test on `ubuntu-latest` for both pushes and pull requests
- Run image push, SonarQube scan, and deployment on the self-hosted ops runner only for pushes to `main`
- Log in to Azure with the runner managed identity
- Read ACR details from Terraform outputs instead of hardcoding them
- Build and push `backend:${GITHUB_SHA}` and `backend:latest`
- Deploy with `ansible-playbook config/ansible/site.yml --limit "localhost,backend_vms"`
- Do not use static Ansible inventory files
- Let Ansible build inventory from `terraform output -json`

Deployment notes:

- The backend deploy reads non-secret topology values from Terraform outputs
- Sensitive values remain in Key Vault
- Set `ANSIBLE_USE_BASTION=false` when the deploy command runs on the ops runner inside the VNet
- `BACKEND_CORS_ALLOWED_ORIGINS` may be supplied as a GitHub Actions variable for custom domains; otherwise deployment defaults to the App Gateway frontend origin from Terraform outputs
- The ops VM managed identity needs `AcrPush` on ACR
